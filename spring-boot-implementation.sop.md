# Spring Boot Implementation SOP

## Overview

This SOP translates the domain decomposition output from `banking-monolith-decomposition.sop.md`
into buildable Spring Boot / Maven modules.

**Core principle: decomposition preserves, not enhances.**
The decomposed services MUST mirror the architecture of the original monolith.
Features absent from the monolith MUST NOT be added during decomposition.

The table below is the result of analysing the actual monolith codebase:

| Feature | Present in monolith | Carry forward? |
|---|---|---|
| Spring Boot Web | Yes | Yes — all services |
| Spring Data JPA | Yes | Yes — all services |
| Spring Security (HTTP Basic) | Yes — `SecurityConfig.java` | Yes — carry forward as-is |
| Bean Validation | Yes | Yes — all services |
| H2 database | Yes (runtime scope) | Yes — keep for local/test |
| Lombok | Yes | Yes — all services |
| Flyway / schema migration | **No** | **No** — do not add |
| OpenAPI / Swagger | **No** | **No** — do not add |
| Resilience4j circuit breakers | **No** | **No** — do not add |
| GlobalExceptionHandler | **No** | **No** — do not add |

If the team later decides to add absent features, that is a separate enhancement task — not part of decomposition.

---

## Parameters

- **monolith_path** (required): Root path of the monolith — used to read `pom.xml` and source files
- **output_path** (required): Path to the decomposed services directory produced by the decomposition SOP
- **db_type** (optional, default: `same-as-monolith`): Target database engine. Defaults to the database detected in the monolith datasource config. Override to change (e.g., `postgresql`, `mysql`)

---

## Steps

### 1. Read Monolith pom.xml

Before generating any file, read `{monolith_path}/pom.xml` and record:
- Spring Boot parent version
- Every dependency `groupId:artifactId:scope`
- Every build plugin

**Constraints:**
- You MUST use the exact Spring Boot version from the monolith — do NOT upgrade or downgrade
- You MUST carry forward every dependency that exists in the monolith
- You MUST NOT add dependencies absent from the monolith unless required to make the decomposed module compile (e.g., a JDBC driver swap for a different `db_type`, or `shared-kernel` which is a new decomposition artifact)

---

### 2. Generate pom.xml for Each Service

Generate a `pom.xml` for each deployable service based on the monolith dependencies.

**Constraints:**
- You MUST include all dependencies carried forward from Step 1
- You MUST add `spring-boot-starter-test` and `spring-security-test` in `test` scope — required to compile and run the unit tests generated by the decomposition SOP
- You MUST add `shared-kernel` as a compile dependency
- You MUST use the JDBC driver matching `{db_type}` — if `db_type` differs from the monolith, replace the driver; otherwise carry it forward as-is
- You MUST NOT add `flyway-core`, `springdoc-openapi`, `resilience4j`, or any other dependency not present in the monolith

### 3. Generate Application Entry Point

**Constraints:**
- You MUST create `{ServiceName}Application.java` in the service's root package
- You MUST declare `scanBasePackages` to include both the service package and `com.bank.shared` so shared-kernel `@Component` beans (e.g., `AuditLogger`) are discovered at startup:

```java
@SpringBootApplication(scanBasePackages = {"com.bank.{service}", "com.bank.shared"})
public class {ServiceName}Application {
    public static void main(String[] args) {
        SpringApplication.run({ServiceName}Application.class, args);
    }
}
```

- You MUST NOT omit `scanBasePackages` — the default scan covers only the service's own package and will fail to find shared beans

---

### 4. Carry Forward SecurityConfig

The monolith has a `SecurityConfig` — it MUST be carried forward to each decomposed service.

**Constraints:**
- You MUST read `{monolith_path}/src/main/java/com/bank/config/SecurityConfig.java` and reproduce it in each service's config package
- You MUST adapt `requestMatchers` to also permit `/internal/events` because this new endpoint must be reachable without authentication from other services
- You MUST NOT add new security patterns (OAuth2, JWT, etc.) that were not in the monolith

Adapted `SecurityConfig` for each decomposed service:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/internal/events").permitAll()
                .requestMatchers("/api/**").authenticated()
                .anyRequest().permitAll()
            )
            .httpBasic(basic -> {});
        return http.build();
    }
}
```

### 5. Generate Configuration Files

**Constraints:**
- You MUST create `src/main/resources/application.yml` for each service containing only the properties relevant to that service's domain — identified from the config inventory in Phase 1 of the decomposition SOP
- You MUST NOT add `spring.flyway.*` — the monolith does not use Flyway
- You MUST NOT add `springdoc.*` — the monolith does not use OpenAPI
- You MUST NOT add `resilience4j.*` — the monolith does not use circuit breakers
- You MUST set `spring.jpa.hibernate.ddl-auto` to match the monolith's value for production, and `create-drop` for local/test profiles
- You MUST create `src/main/resources/application-local.yml` overriding the datasource to H2 in-memory so the service starts locally without a running database server
- You MUST create `src/test/resources/application.properties` with H2 datasource and `ddl-auto=create-drop` so tests run without infrastructure:

```properties
spring.jpa.hibernate.ddl-auto=create-drop
spring.datasource.url=jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1
spring.datasource.driver-class-name=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=
```

---

### 6. Generate JPA Entities

**Constraints:**
- You MUST generate exactly one `@Entity` class per database table per service — never two classes mapped to the same `@Table(name=...)` because duplicate mappings cause `BeanCreationException` at startup
- You MUST copy all `enum` types and their constants verbatim from the monolith source — read the actual source file; do NOT infer or substitute values
- You MUST replace all cross-domain `@ManyToOne` / `@OneToMany` JPA relationships with plain scalar reference columns (e.g., `private Long customerId` instead of `@ManyToOne Customer customer`)
- You MUST add `@JsonIgnore` on any remaining bidirectional relationship fields to prevent circular JSON serialisation
- You MUST use Lombok `@Data` consistent with the monolith's existing style

---

### 7. Test Setup

**Constraints:**
- You MUST annotate every `@DataJpaTest` class with `@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.ANY)` and `@TestPropertySource(properties = {"spring.jpa.hibernate.ddl-auto=create-drop"})` because `@DataJpaTest` does not reliably pick up `application.properties`
- You MUST annotate every `@WebMvcTest` class with `@AutoConfigureMockMvc(addFilters = false)` and `@WithMockUser` to bypass Spring Security in unit tests — without these, all requests return 401/403
- You MUST construct any service class that takes `Optional<T>` constructor parameters manually in `@BeforeEach` rather than using `@InjectMocks`
- You MUST mock `RestTemplate` in `EventPublisher` and cross-domain client tests — do not make real HTTP calls

---

## Troubleshooting

### Service fails to start — "connection refused" to database
Start with `--spring.profiles.active=local` to use the H2 in-memory profile.

### @DataJpaTest fails with "missing table"
Add `@AutoConfigureTestDatabase(replace=ANY)` and `@TestPropertySource(properties={"spring.jpa.hibernate.ddl-auto=create-drop"})` directly on the test class.

### @WebMvcTest returns 401 or 403
Add `@AutoConfigureMockMvc(addFilters = false)` and `@WithMockUser`. Ensure `spring-security-test` is in `test` scope.

### @InjectMocks fails for service with Optional constructor parameter
Construct the service manually: `service = new MyService(repo, logger, Optional.of(mockPublisher));`

### Missing @SpringBootApplication causes test context failure
Create `{ServiceName}Application.java` with `@SpringBootApplication(scanBasePackages={"com.bank.{service}","com.bank.shared"})` in the service's root package.

### No qualifying bean of type 'AuditLogger' available
Add `scanBasePackages = {"com.bank.{service}", "com.bank.shared"}` to the application class.

### Duplicate @Entity for same table causes BeanCreationException
Two `@Entity` classes are mapped to the same `@Table(name=...)`. Pick one package (`domain` or `model`) and delete the other.
